# Workshop Release-Candidate Acceptance Gate

## Purpose

This document defines the final release-candidate acceptance gate for the Browser-Safe AI Systems workshop lab track.

The gate gives a technically unforgiving reviewer one local decision point before the workshop is treated as release-candidate material for classroom use. It ties together the offline classroom bundle, release rehearsal evidence, lab coverage matrix, reviewer rubric, safety boundary, synthetic marker checks, and instructor readiness checklist.

This gate is local-only, synthetic-only, and authorized-only. It does not claim production security validation.

## Scope

The acceptance gate applies to the repository state being reviewed and the evidence generated by:

```text
tools/build_workshop_offline_release_bundle.py
tools/run_workshop_release_rehearsal.py
tools/run_workshop_release_candidate_acceptance_gate.py
```

The gate covers:

```text
offline release bundle build and verifier execution
release rehearsal evidence generation and validation
Lab 00 through Lab 12 document coverage
lab coverage matrix consistency
reviewer grading rubric completeness
safety boundary preservation
SYNTHETIC-LAB-MARKER coverage for synthetic adversarial fixtures
instructor readiness checklist generation
release-candidate evidence archive production
clear pass, fail, or needs-owner decision criteria
```

The gate does not test third-party systems, public callback endpoints, production SaaS tenants, real user data, real credentials, real cookies, real tokens, malware, browser command and control, or real customer data.

## Tool

Run the release-candidate acceptance gate from the repository root:

```bash
.venv/bin/python tools/run_workshop_release_candidate_acceptance_gate.py \
  --repo-root /home/foo/Workspace/ai-browser-security-test-suite \
  --out-dir "$HOME/browser-safe-ai-workshop-development-evidence/slice-2.0-workshop-release-candidate-acceptance-gate/release-candidate-gate-$(date -u +%Y%m%d-%H%M%S)" \
  --python-bin /home/foo/Workspace/ai-browser-security-test-suite/.venv/bin/python \
  --bundle-stem browser-safe-ai-workshop-offline-release-bundle
```

The helper runs the release rehearsal helper unless an existing rehearsal evidence directory is supplied with `--release-rehearsal-dir`.

## Acceptance decision values

Use only these decision values:

| Decision | Meaning |
| --- | --- |
| `pass` | Every mandatory gate check passed and the reviewer can trace evidence, safety, coverage, release rehearsal, and instructor readiness locally. |
| `fail` | One or more mandatory checks failed or the evidence exceeds the declared local synthetic scope. |
| `needs-owner` | The reviewer cannot decide because required evidence, command output, archive identity, or explanation is unavailable. |

Do not use informal alternatives such as "looks good", "probably fine", or "good enough".

## Gate 1, repository identity and worktree state

The reviewer should mark this gate `pass` only when:

```text
repository root is recorded
repo HEAD is recorded
current branch is recorded
worktree is clean before final gate evidence is generated
```

A dirty worktree during final release-candidate evidence generation is a `fail` unless the reviewer is explicitly running a pre-commit dry run. A missing repository identity is `needs-owner`.

## Gate 2, offline release bundle

The reviewer should mark this gate `pass` only when the release rehearsal evidence proves:

```text
offline release bundle archive was built
offline release archive SHA256 verified
offline release bundle was extracted
VERIFY_BUNDLE.sh passed with the prepared Python interpreter
bundled validators passed
bundled pytest passed
```

The offline release bundle must include:

```text
docs/workshop/
docs/workshop/release-candidate-acceptance-gate.md
docs/schemas/
docs/target-contracts/
examples/
payloads/
src/
tools/
tests/
VERIFY_BUNDLE.sh
RUN_OFFLINE_PREFLIGHT.sh
offline-release-manifest.json
SHA256SUMS.txt
```

A missing verifier result is `fail`. A missing archive is `needs-owner`.

## Gate 3, release rehearsal evidence

The reviewer should mark this gate `pass` only when the release rehearsal evidence directory contains:

```text
release-rehearsal-plan.json
release-rehearsal-timing.json
release-rehearsal-report.md
instructor-release-rehearsal-checklist.md
release-rehearsal-command-log.jsonl
release-rehearsal-artifact-manifest.json
SHA256SUMS.txt
offline-bundle/
offline-bundle-extract/
```

The reviewer should verify `SHA256SUMS.txt` inside the rehearsal evidence directory and confirm that `release-rehearsal-report.md` states the local-only, synthetic-only, authorized-only boundary.

## Gate 4, lab coverage matrix consistency

The reviewer should mark this gate `pass` only when:

```text
docs/lab-track-coverage-matrix.md exists
Lab 00 through Lab 12 are listed
the matrix separates implemented workshop labs from future gaps
the matrix preserves local-only, synthetic-only, authorized-only scope
the matrix does not claim production security validation
```

The matrix may still identify future hardening work. Future work does not block release-candidate status unless it contradicts the acceptance scope or hides a required classroom dependency.

## Gate 5, reviewer rubric completeness

The reviewer should mark this gate `pass` only when `docs/workshop/reviewer-grading-rubric.md` includes:

```text
evidence quality criteria
safety boundary criteria
capstone review expectations
fail conditions
local-only, synthetic-only, authorized-only scope
no real credentials
no real customer data
no third-party target testing
no production security claims
```

A rubric that allows passing a student without evidence artifacts is a `fail`.

## Gate 6, safety boundary and non-claims

The reviewer should mark this gate `pass` only when the documentation and generated report preserve all of these statements:

```text
local-only
synthetic-only
authorized-only
no real credentials
no real customer data
no real cookies
no real tokens
no public callback endpoints
no production SaaS tenants
no third-party AI products
no malware
no browser command and control
no production security validation claim
```

Any broad assurance statement that exceeds the evidence scope is a `fail`.

## Gate 7, synthetic marker coverage

The reviewer should mark this gate `pass` only when `SYNTHETIC-LAB-MARKER` remains present in the synthetic adversarial fixture documentation and first-party fixture generators.

At minimum, the gate checks:

```text
docs/workshop/labs/02 through 12 where synthetic adversarial fixtures are documented
tools/generate_lab_02_indirect_prompt_fixtures.py
tools/generate_lab_03_hidden_dom_fixtures.py
tools/generate_lab_04_dom_render_mismatch_fixtures.py
tools/generate_lab_05_screenshot_visual_deception_fixtures.py
tools/generate_lab_07_delayed_content_state_transition_fixtures.py
tools/generate_lab_08_qr_handoff_fixtures.py
tools/generate_lab_09_synthetic_sensitive_data_fixtures.py
tools/generate_lab_10_model_verdict_policy_fixtures.py
tools/generate_lab_11_fail_open_exception_fixtures.py
tools/generate_lab_12_capstone_evidence_package.py
```

A synthetic fixture that removes the marker or substitutes real credentials, real customer data, real brands as live impersonation targets, public callback endpoints, malware, token theft, credential theft, MFA bypass, or browser command and control is a `fail`.

## Gate 8, instructor readiness

The reviewer should mark this gate `pass` only when the generated `instructor-readiness-checklist.md` records:

```text
offline bundle verifier passed
release rehearsal evidence exists
Lab 00 through Lab 12 are present
safety boundary statement is ready for delivery
reviewer rubric is available
student evidence expectations are clear
troubleshooting document is available
known limitations are stated
```

The instructor checklist is not a substitute for a live classroom timing rehearsal on the target classroom hardware. Missing classroom timing calibration can remain a documented limitation if the release-candidate claim is limited to repository and release-package readiness.

## Gate 9, evidence archive production

The reviewer should mark this gate `pass` only when the release-candidate gate produces:

```text
release-candidate acceptance evidence directory
release-candidate acceptance evidence .tar.gz
release-candidate acceptance evidence .tar.gz.sha256
SHA256SUMS.txt for generated artifacts
artifact manifest with SHA256 entries
```

The evidence archive must not be stored inside the repository tree.


The reviewer should not record `pass` unless every mandatory gate above is satisfied.

## What this gate proves

If all gates pass, the reviewer can say:

```text
For the reviewed repository state, the workshop release-candidate package can be built, hashed, extracted, verified, rehearsed, checked against the lab coverage matrix, checked against the reviewer rubric, checked for safety-boundary language, checked for synthetic markers, and archived as local reviewer evidence.
```

## What this gate does not prove

This gate does not prove production browser-AI security.

This gate does not prove:

1. Production browser-AI security.
2. Safety of any third-party browser-AI product.
3. Real credential, token, cookie, or session protection.
4. Real customer-data protection.
5. Tenant isolation.
6. Browser extension isolation.
7. That model output can make policy decisions.
8. That all Browser-Safe AI Systems series risks now have executable coverage.
9. That the intentionally weak local `ollama-webui` target is hardened.
10. That classroom timing is calibrated for every venue or hardware profile.

## Required release-candidate status

Release-candidate status is appropriate only when the generated report says:

```text
overall decision: pass
```

Any `fail` result blocks release-candidate status.

Any `needs-owner` result blocks release-candidate status until the missing evidence or explanation is supplied.
